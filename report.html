Implementation: Define a Performance Metric¶

It is difficult to measure the quality of a given model without quantifying its performance over training and testing. This is typically done using some type of performance metric, whether it is through calculating some type of error, the goodness of fit, or some other useful measurement. For this project, you will be calculating the coefficient of determination, R2, to quantify your model's performance. The coefficient of determination for a model is a useful statistic in regression analysis, as it often describes how "good" that model is at making predictions.

The values for R2 range from 0 to 1, which captures the percentage of squared correlation between the predicted and actual values of the target variable. A model with an R2 of 0 is no better than a model that always predicts the mean of the target variable, whereas a model with an R2 of 1 perfectly predicts the target variable. Any value between 0 and 1 indicates what percentage of the target variable, using this model, can be explained by the features. A model can be given a negative R2 as well, which indicates that the model is arbitrarily worse than one that always predicts the mean of the target variable.

Answer:

  • To explain what k-fold cross validation, it's better to first explain cross validation. Corss validation means after we split the all the sample data sets into training, validation and testing datasets, we use training dataset for training model, use validation dataset to tune our model, and use testing dataset to finally test how good our model is. In cross validation, we drastically reduce the number of samples that can be used to train our model, and moreover, the result highly depends on the a particular random choice of pair (train, validation). The k-fold cross validation is an approach that the model can learn from all of the sample dataset (of course except for the testing data which we need to reserve at very beginning), and also doesn't fall into the "overfitting" trap. In k-fold cross validation, we again need to hold out testing dataset first. Then we break the entire remaining dataset into k buckets, and we train our model k rounds, and in each round, we use the data in one bucket (different each round) as the validation dataset, and we use the remaining data as the training dataset. In each round, we can test our model use this particular train/test pair. The final performance of our model will be averaged over rounds. And finally, we use our testing dataset to judge our model's performance.

  • Grid search still uses particular pair of (training, testing) from the sample dataset, which will have almost the same issues as cross validation, mentioned above. So we can incorporate k-fold cross validation with grid search to help solve this issue. It will be like a nested k-fold cross validation process. In order to do this, first we still need to draw a "grid system" with combinations of different hyper-parameters we think might work, then for each grid cell (for each particular combination), do the k-fold cross validation process, to see how good our model is using this particular combination of hyper-parameters. Repeat until we have tested each grid cell, and find the best model in this case.

from sklearn.metrics import make_scorer
from sklearn.tree import DecisionTreeRegressor
from sklearn.model_selection import GridSearchCV

def fit_model(X, y):
    """ Performs grid search over the 'max_depth' parameter for a 
        decision tree regressor trained on the input data [X, y]. """
    
    # Create cross-validation sets from the training data
    # sklearn version 0.18: ShuffleSplit(n_splits=10, test_size=0.1, train_size=None, random_state=None)
    # sklearn versiin 0.17: ShuffleSplit(n, n_iter=10, test_size=0.1, train_size=None, random_state=None)
    cv_sets = ShuffleSplit(n_splits = 10, test_size = 0.20, random_state = 0)

    # TODO: Create a decision tree regressor object
    regressor = DecisionTreeRegressor()

    # TODO: Create a dictionary for the parameter 'max_depth' with a range from 1 to 10
    dt_range = range(1, 11)
    params = dict(max_depth=dt_range)
    
    # TODO: Transform 'performance_metric' into a scoring function using 'make_scorer' 
    scoring_fnc = make_scorer(performance_metric)

    # TODO: Create the grid search cv object --> GridSearchCV()
    # Make sure to include the right parameters in the object:
    # (estimator, param_grid, scoring, cv) which have values 'regressor', 'params', 'scoring_fnc', and 'cv_sets' respectively.
    grid = GridSearchCV(regressor, params, cv=cv_sets, scoring=scoring_fnc)

    # Fit the grid search object to the data to compute the optimal model
    grid = grid.fit(X, y)

    # Return the optimal model after fitting the data
    return grid.best_estimator_

Question 11 - Applicability¶

  • In a few sentences, discuss whether the constructed model should or should not be used in a real-world setting.

Hint: Take a look at the range in prices as calculated in the code snippet above. Some questions to answering:

  • How relevant today is data that was collected from 1978? How important is inflation?
  • Are the features present in the data sufficient to describe a home? Do you think factors like quality of apppliances in the home, square feet of the plot area, presence of pool or not etc should factor in?
  • Is the model robust enough to make consistent predictions?
  • Would data collected in an urban city like Boston be applicable in a rural city?
  • Is it fair to judge the price of an individual home based on the characteristics of the entire neighborhood?

Answer:

  • Inflation is a big concern here, and that makes the house price collected in 1978 not completely consistent with the house price in Boston today. For example, according to the dataset we used in this project, the median house price is 438,900 USD. But today, the median house price is around 560,000 USD. That is about 27% increase, and this is not something neglectable.
  • It is likely the three features we used in this project are still insufficient to describe a home. In fact, when people are buying houses, the biggest concern is how large the house is, which means the floor area (and number of floors) is an important feature to be listed in. Moreover, houses are built in different time, so the house age is also an important feature (newer house tends to have better infrastructure and appliances). And of course, if the house is equipped with a pool or similar luxury components, it is always selled at higher price compared with the house without them.
  • To some extent, the model is relatively robust, but not very robust. For example, when predicting the house price for client 1 for 10 times above, the fluctuation of the result (69,044 USD) is about 17% of the predicted price (around 400,000 USD). This fluctuation is ok (not too large, but not too small as well).
  • The house price data collected in urban area is very likely not applicable at rural area. Rural area is normally less developed than urban areas, so the living expenses in rural area is also less then that in city. Moreover, houses (or buildings in general) in rural areas are likely constructed in different material or style (terrace, detached, etc). These issues mean that predicting house price in rural areas is not the same as that in city.
  • In my opinion, judging the house prices on the entire neighbourhood is not considerate enough, since different neighbourhood can have different number of houses (let's say one neighbourhood has 10 houses, and another has 100, for example). It is likely more feasible to only consider the fixed number of houses (let's say 20 for example) that are nearest to the house we are judging. I mean, it looks more fair, that each house is judged by same number of houses near it. But of course, collecting data this way, is much more difficult that just getting the neighbourhood data, because now we need to deal with smaller trunk of areas.
